# AIDataInsight-iOS 架构设计说明书

## 1. 文档目的

本文档基于当前仓库的实际实现，说明 AIDataInsight-iOS 的工程组织方式、模块职责、核心数据流和关键设计点，重点覆盖 `ModuleAI` 聊天分析模块以及新增的流式响应能力。

## 2. 总体架构

项目采用“App 壳工程 + Swift Package 模块”的组织方式：

```text
AIDataInsight App
    -> 业务模块（ModuleAI）
    -> 通用业务层（LibraryCommon）
    -> 基础设施层（LibraryBasics）
```

### 2.1 架构目标

- 模块解耦：业务能力、通用业务能力、基础设施分层清晰
- 易维护：业务代码尽量收敛到对应 package 内部
- 易扩展：AI 场景、图表场景、账号场景可独立演进
- 可验证：网络请求、ViewModel、页面交互具备清晰边界
- 体验友好：支持 AI 文本流式输出和图表反馈

### 2.2 架构风格

当前项目以 **UIKit + MVVM（Closure 回调）** 为主：

- `ViewController`：负责页面生命周期、视图层级、用户交互、列表刷新
- `ViewModel`：负责业务编排、接口调用、模型转换、状态输出
- `Model`：承载聊天消息、函数调用结果、图表数据、历史记录等实体

与常见响应式 MVVM 不同，当前项目没有引入 Combine，而是使用闭包回调配合 `BaseViewModel` 的任务管理。

## 3. 目录与模块划分

## 3.1 App 壳工程

路径：`AIDataInsight/AIDataInsight`

职责：

- 应用生命周期入口
- Scene / Window 装配
- `Info.plist`、`entitlements`
- 环境配置 `Xcconfig`
- 集成 Swift Package 产物

壳工程自身业务逻辑很轻，核心功能基本在 `Packages` 中实现。

## 3.2 业务模块：ModuleAI

路径：`Packages/module-ai`

职责：

- AI 数据分析聊天入口
- 欢迎语、历史记录、用户输入、AI 回复
- 函数调用分析与意图识别
- 查询结果的图表渲染
- 流式文本回复

核心文件：

- `AIChat/AIChatViewController.swift`
- `AIChat/ViewModels/AIChatViewModel.swift`
- `AIChat/Models/AIChat.swift`
- `AIChat/Models/FunctionModel.swift`
- `AIChat/Models/HistoryDetailModel.swift`
- `Api/ChatApi.swift`
- `Api/HistoryApi.swift`
- `Api/ChartApi.swift`

### 3.2.1 AIChatViewController

职责：

- 组织聊天页面 UI
- 使用 `UICollectionViewDiffableDataSource` 管理消息列表
- 响应底部输入框发送事件
- 驱动欢迎消息、历史消息、图表消息、错误消息展示
- 负责 AI 流式消息的 UI 层节流渲染

### 3.2.2 AIChatViewModel

职责：

- 拉取历史详情
- 加载模板问题
- 发起函数调用分析
- 发起图表查询
- 发起流式聊天请求
- 将网络返回映射成页面可消费的闭包输出

### 3.2.3 AIChat 相关 Cell

当前聊天列表根据消息类型渲染不同 Cell：

- `AIChatWelcomeCell`
- `AIChatUserCell`
- `AIChatCell`
- `AIChatIntentCell`
- `AIChatChartCell`
- `AIChatLegendChartCell`

这种设计把不同表现形态隔离开，便于后续新增消息类型。

## 3.3 通用业务层：LibraryCommon

路径：`Packages/library-common`

职责：

- 提供通用业务层入口
- 统一请求封装
- 模块协议定义
- 通用 ViewModel 能力承接

核心组件：

- `CommonViewModel`
- `CommonRequester`
- `ProtocolAI`
- `LoginProtocol`
- `PrivacyProtocol`
- `SettingProtocol`

### 3.3.1 CommonRequester

`CommonRequester` 对上层业务提供统一请求方式：

- `request`
- `requestNet`
- `requestVoid`
- `requestSSE`

前 3 类请求用于常规接口，`requestSSE` 用于流式响应场景。

## 3.4 基础设施层：LibraryBasics

路径：`Packages/library-basics`

职责：

- 网络基础能力
- UI 基础组件
- ViewModel 基类
- 路由、环境、账号等技术基础设施

与本项目当前主链路直接相关的组件：

- `Networking`
- `SSEClient`
- `BaseViewModel`
- `BaseUI`
- `BaseKit`
- `Router`

### 3.4.1 Networking

网络层基于 `Moya` 风格封装，项目内业务接口统一实现 `CustomTargetType`，提供：

- `baseURL`
- `path`
- `method`
- `parameters`
- `headers`

AI 场景中的 `ChatApi`、`HistoryApi`、`ChartApi` 都遵循该规范。

### 3.4.2 SSEClient

`SSEClient` 当前位于 `library-basics/Sources/Networking`。从项目分层上看，这是更合理的位置，因为它本质上是纯网络传输能力，而不是 `CommonViewModel` 独有的业务通用逻辑。

职责：

- 基于 `URLSession + dataTask` 建立流式连接
- 校验响应状态码
- 解析标准 SSE `data:` 事件
- 将事件文本逐条回调给上层
- 支持取消与结束回调

放在 `Networking` 的原因：

- 便于多个业务模块复用
- 避免 `library-common` 承担底层网络实现职责
- `CommonRequester` 仍可保留统一请求入口角色，不影响上层调用

### 3.4.3 BaseViewModel

`BaseViewModel` 提供统一任务管理：

- `trackTask`
- `cancelTask`
- `cancelAllTasks`

这让页面退出或切换会话时，可以安全取消正在进行的网络任务，包括 SSE 流式任务。

## 4. AI 聊天核心数据流

## 4.1 历史记录加载

```text
AIChatViewController
-> AIChatViewModel.getHistoryDetail
-> HistoryApi.detail
-> CommonRequester.requestNet
-> RecordModel / DetailModel
-> 映射为 AIChat
-> UICollectionView 刷新
```

## 4.2 模板问题加载

```text
AIChatViewController
-> AIChatViewModel.loadTemplate
-> ChatApi.template
-> CommonRequester.requestNet
-> TemplateModel
-> 欢迎消息渲染
```

## 4.3 函数调用分析链路

```text
用户发送消息
-> AIChatViewController.sendUserMessage
-> AIChatViewModel.sendFunctionMessage
-> ChatApi.function
-> FunctionModel
-> 意图识别 / 参数抽取
-> ChartApi.chart
-> HistoryDetailModel
-> 图表数据转换
-> 图表 Cell 渲染
```

特点：

- 适用于分析型问题
- 返回结果可能是意图补充提示，也可能是图表数据
- 由 ViewModel 承担编排职责，ViewController 不直接参与业务判断

## 4.4 流式聊天链路

```text
用户发送消息
-> AIChatViewController.sendUserMessage
-> AIChatViewModel.sendStreamMessage
-> CommonRequester.requestSSE
-> SSEClient
-> onStreamText(chunk)
-> pendingStreamText 累积
-> CADisplayLink 节流
-> renderedStreamText 增量推进
-> AIChatCell 局部刷新
```

## 5. 流式渲染设计

## 5.1 设计动机

单纯把 SSE chunk 到达就立刻渲染到 UI，会有两个问题：

- mock / 真服务端返回过快时，看不到打字机效果
- 列表 item 高度频繁变化时，容易出现抖动

因此当前实现把“网络接收速度”和“UI 显示速度”拆开处理。

## 5.2 双缓冲文本状态

在 `AIChatViewController` 中维护两份文本：

- `pendingStreamText`：已经从网络收到的完整待显示文本
- `renderedStreamText`：已经实际展示到 UI 的文本

网络回调只负责追加到 `pendingStreamText`，不直接刷新 UI。

## 5.3 CADisplayLink 节流

通过 `CADisplayLink` 驱动显示层节奏：

- 每帧只推进一小段文本
- 即使后端瞬间返回全文，也能继续做逐步显示
- 流式完成和 UI 显示完成分离，避免一结束就跳到全文

## 5.4 英文单词边界推进

为了让英文显示更自然，当前推进策略不是简单按字符截断，而是：

- 英文和数字连续段按“单词”整体推进
- 中文和其它非 ASCII 文本按字符推进
- 空格会跟随英文单词一起进入，避免单词间抖动

这样可以避免 `hel` / `hell` / `hello` 这种生硬断裂。

## 5.5 局部刷新与防抖动

当前实现优先更新最后一个可见 AI 文本 Cell，而不是每次都重建整个 diffable snapshot。

换行导致的高度变化使用：

- `cell.configure(with:)`
- `collectionView.collectionViewLayout.invalidateLayout()`
- `collectionView.layoutIfNeeded()`
- 无动画滚动到底部

这样可以降低 `performBatchUpdates(nil)` 带来的高度插值动画和跳动问题。

最终流式完成后，再将最终文本回写到 snapshot，保证数据源一致性。

## 6. 网络与任务管理

## 6.1 常规请求

常规接口统一走 `CommonRequester.requestNet`，由 `ResponseModel<T>` 解析响应，回到主线程后通知页面。

## 6.2 流式请求

流式请求统一走 `CommonRequester.requestSSE`：

- 创建 `SSEClient`
- 绑定 `ClosureCancellableTask`
- 向上层输出 `onEvent`
- 在结束时输出 `completion`

## 6.3 取消机制

基于 `BaseViewModel.trackTask`，SSE 任务可被统一管理。当前在以下场景会主动取消：

- 清空聊天
- 页面释放
- 新任务覆盖旧任务

## 7. 设计取舍

## 7.1 为什么保留 UIKit + Closure MVVM

当前项目已经形成 UIKit 页面体系和模块边界，继续沿用可以保证：

- 改动范围可控
- 页面接入成本低
- 与现有基础库风格一致

## 7.2 为什么 SSE 放在 LibraryBasics / Networking

SSE 是通用通信能力，不属于 `ModuleAI` 私有实现，也不应依附在 `CommonViewModel` 这种业务通用层上。放在 `LibraryBasics/Networking` 更合理：

- 更符合“基础传输能力归基础设施层”的边界
- 便于其它模块直接复用
- `CommonRequester` 仍可作为上层统一入口，不影响业务调用方式
- 避免 `library-common` 同时承担业务包装和底层网络实现两种职责

## 7.3 为什么不在每个 chunk 到达时直接刷新列表

因为直接刷新会导致：

- 高频 UI 更新
- diffable data source 反复 apply
- 列表 item 换行时明显抖动

因此当前采用“网络快收、UI慢吐”的设计。

## 8. 可扩展方向

- 将 SSE 请求头、鉴权、超时策略收敛到统一配置
- 将流式消息抽象成通用消息协议，支持更多消息类型
- 为 AI 回复增加 Markdown / 富文本渲染能力
- 将当前 Closure 回调逐步迁移到 Swift Concurrency
- 增加针对 ViewModel 与 SSEClient 的自动化测试

## 9. 总结

AIDataInsight-iOS 当前已经具备以下比较清晰的结构特征：

- App 壳工程负责装配
- `ModuleAI` 承担 AI 数据分析业务
- `LibraryCommon` 承担通用业务请求入口与协议层
- `LibraryBasics` 提供网络、SSE、基类、UI 和工具能力

在聊天场景中，项目同时支持：

- 函数调用驱动的分析型问答
- SSE 驱动的流式文本回复

这使项目既能展示 AI 业务编排，也能覆盖面向用户的实时交互体验。
